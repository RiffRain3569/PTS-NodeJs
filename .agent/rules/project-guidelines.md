# 프로젝트 규칙 및 코드 스타일 가이드 (Project Guidelines)

이 문서에는 사용자가 지정한 전역 코드 스타일 및 작업 규칙이 포함되어 있습니다. 코드를 작성하거나 수정할 때, 또는 질문에 답변할 때 항상 이 규칙을 엄격하게 적용하십시오.

## 기본 원칙 (General Rules)
- 모든 답변 및 커밋 메시지는 반드시 **한글(Korean)** 로 작성합니다.
- 토이 프로젝트 수준의 예제가 아닌 **실제 프로덕션 환경(production usage)** 을 가정하고 코드를 작성합니다.
- 숨겨진 동작이나 매직 코드보다는 **명시적인 접근(Favor explicitness)** 을 선호합니다.
- 장기적인 **유지보수성(long-term maintainability)** 을 최적화해야 합니다.

## 백엔드 설계 가이드 (Backend Design Rules)
- **Controller는 최대한 얇게 유지**합니다 (Keep controllers thin).
- **비즈니스 로직**은 반드시 Service 또는 Domain 계층에 배치합니다.
- 인프라스트럭처의 관심사가 Domain 로직으로 유출되지 않도록 분리합니다.
- 합리적인 선에서 **불변성(Immutability)** 을 선호합니다.
- API는 항상 **예측 가능**해야 하며, **버전 관리(Versionable)** 가 가능하도록 설계합니다.

## 에러 처리 (Error Handling)
- 프로젝트 전체에서 **일관된 에러 응답 구조(Consistent error response structures)** 를 사용합니다.
- 예외적인 상황이나 엣지 케이스(Edge cases)를 절대 무시하지 않습니다.
- 안정성이나 보안에 영향을 미치는 설계 결정사항이라면 **항상 그 이유(Why)를 명시적으로 설명**합니다.

## 보안 (Security)
- 사용자의 입력을 신뢰 가능한 것으로 **절대 가정하지 않습니다** (Never assume trusted input).
- 관련된 경우 유효성 검사(Validation), 인증(Authentication), 인가(Authorization)를 반드시 고려하고 반영합니다.

## 응답 스타일 (Output Style)
- 답변은 간결하지만 완전해야 합니다 (Concise but complete).
- 이모티콘(Emojis)은 **절대 사용하지 않습니다**.
- 불필요하고 중복된 설명은 생략합니다.

## Git 커밋 규칙 (Git Commit Rules)
- 반드시 **Conventional Commits** 사양을 따릅니다.
- **포맷**: `<type>(<scope>): <subject>`

### 세부 커밋 작성 가이드:
- **주제 (Subject)**:
  - 간결하고 명확하게 **한글**로 작성합니다.
  - 현재 시제를 사용합니다.
  - 문장 끝에 마침표(.)를 찍지 않습니다.
  - "수정", "변경", "업데이트"와 같이 모호한 단어는 사용하지 않습니다.
  - 단순한 동작(Action)을 나열하는 대신, **변경의 의도와 이유**를 설명합니다.
- **범위 (Scope)**:
  - 의미를 가진 도메인 이름이나 모듈 이름을 사용합니다 (예: `auth`, `bithumb-order`).
  - 전역적인 변경 사항이 아닌 이상 **범위를 생략하지 않습니다**.
- **본문 (Body)**:
  - 꼭 필요한 경우에만 작성합니다.
  - 작성 시 **한글**로 서술하며, 글머리 기호(Bullet points)를 사용합니다.
  - *어떻게(how)* 가 아니라 **왜(why)** 변경했는지 그 이유를 설명합니다.
  - 이모티콘은 절대 포함하지 않습니다.
  - 커밋 메시지 외에 불필요한 텍스트를 적지 않습니다.
